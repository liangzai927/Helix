# Helix Documentation Checklist

## Minimum Truthfulness Checks

- Does every referenced path exist now
- Does every referenced script or command exist now
- Did you mark planned capabilities as planned
- Did you avoid implying CLI, desktop, or marketplace support before it exists

## README Coverage

Prefer this order:

1. What Helix is
2. What is currently in the repo
3. Core positioning and MVP intent
4. Current structure
5. Authoritative docs
6. Verified usage or next steps

## Release-facing Docs

Before claiming release readiness, confirm the relevant phase:

- README complete
- CHANGELOG complete
- LICENSE present
- icon and screenshots ready when needed
- packaging or install steps actually tested

## Comment Quality

Good comments:

- explain architecture boundaries
- explain why approval or planning exists
- explain non-obvious constraints in adapters, tools, or caches

Avoid comments that merely restate the code.
