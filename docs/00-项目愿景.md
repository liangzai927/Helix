# Helix Agent 项目愿景 / Project Vision

## 1. Helix Agent 是什么 / What Helix Agent Is

### 中文

Helix Agent 不是一个普通的聊天插件。

它的目标是成为一个运行在本地开发环境中的 Agent Engine，让用户在 VS Code 中完成从理解代码、制定计划、生成补丁到执行受控命令的完整闭环。

第一版的交互入口是 VS Code，但 VS Code 只是适配层，不应该承载核心业务逻辑。

### English

Helix Agent is not a regular chat plugin.

Its goal is to become an Agent Engine that runs in the local development environment and enables users to complete a full loop inside VS Code, from code understanding and planning to patch generation and controlled command execution.

The first client is VS Code, but VS Code is only an adapter layer and should not contain the core business logic.

## 2. 核心定位 / Core Positioning

### 中文

Helix Agent 的产品定位是：

- 一个运行在 VS Code 中的独立代码 Agent 产品
- 一个可以读取工程、搜索代码、生成计划、输出补丁、执行受控命令的本地 Agent Engine
- 一个核心能力可复用于 CLI 和 Electron 的分层系统

这意味着项目从第一天开始就要坚持一个边界：

- Agent Core 不依赖 VS Code API
- VS Code 负责 UI、宿主能力接入和审批交互
- 后续 CLI、Electron 复用同一套核心 Runtime、Tool、Model Adapter 能力

### English

Helix Agent is positioned as:

- an independent coding agent product that runs inside VS Code
- a local Agent Engine that can read projects, search code, generate plans, produce patches, and execute controlled commands
- a layered system whose core capabilities can be reused by CLI and Electron

This means the project must keep the following boundary from day one:

- Agent Core must not depend on the VS Code API
- VS Code is responsible for UI, host integration, and approval interactions
- future CLI and Electron clients should reuse the same Runtime, Tool, and Model Adapter capabilities

## 3. 解决的问题 / Problems It Solves

### 中文

Helix Agent 重点解决真实开发中的几个问题：

- 陌生仓库理解成本高
- 多文件修改容易漏改和误改
- 普通聊天式 Copilot 缺少计划和执行闭环
- 长会话上下文越来越重，token 消耗越来越高
- 写文件和执行命令缺少统一的审批与审计机制

Helix Agent 的目标不是“更会聊天”，而是“更适合真实开发执行”。

### English

Helix Agent focuses on several real development problems:

- high cost of understanding unfamiliar repositories
- easy-to-miss or incorrect edits across multiple files
- lack of a planning and execution loop in ordinary chat-style copilots
- increasingly heavy context and token cost in long sessions
- lack of a unified approval and audit mechanism for file edits and command execution

Its goal is not to be "better at chatting," but to be "better at real development execution."

## 4. 第一版产品原则 / First Version Principles

### 4.1 规划优先 / Planning First

中文：复杂任务先探索，再规划，再执行，避免模型直接进入高风险改动。  
English: Complex tasks should be explored first, then planned, then executed, instead of jumping directly into risky edits.

### 4.2 工具优先 / Tools First

中文：Agent 的能力通过结构化工具提供，而不是靠自由发挥的长 Prompt。  
English: Agent capabilities should be provided through structured tools rather than uncontrolled long prompts.

### 4.3 安全可控 / Safe and Controllable

中文：所有高风险动作必须可见、可审查、可中断、可确认。  
English: All high-risk actions must be visible, reviewable, interruptible, and confirmable.

### 4.4 Runtime 与 UI 解耦 / Runtime and UI Decoupling

中文：VS Code 只是第一版客户端，核心 Runtime 必须能独立演进，并为 CLI 和 Electron 复用做好边界约束。  
English: VS Code is only the first client. The core Runtime must evolve independently and preserve clean boundaries for future CLI and Electron reuse.

### 4.5 成本优先 / Cost Efficiency First

中文：产品设计要关注上下文效率、缓存命中率和长会话稳定性，而不是单纯追求单轮回答效果。  
English: The product should prioritize context efficiency, cache hit rate, and long-session stability rather than only optimizing single-turn answer quality.

## 5. 第一版交付目标 / First Version Delivery Goal

### 中文

在 MVP 阶段，Helix Agent 需要完成一个最小但闭环的使用路径：

1. 用户在 VS Code 中发起任务。
2. Agent 读取工程上下文并执行必要的只读探索。
3. Agent 输出可读的执行计划。
4. Agent 生成补丁或发起受控命令。
5. 用户审查后确认执行。
6. Agent 返回修改结果和摘要。

### English

During the MVP stage, Helix Agent should deliver a minimal but complete workflow:

1. The user starts a task inside VS Code.
2. The agent reads project context and performs necessary read-only exploration.
3. The agent outputs a readable execution plan.
4. The agent generates a patch or requests a controlled command execution.
5. The user reviews and approves the action.
6. The agent returns the result and a summary.

## 6. 架构方向 / Architecture Direction

### 中文

第一版按两层理解最清晰：

- Agent Core：负责任务运行时、规划、执行、工具调度、上下文治理、模型适配
- VS Code Adapter：负责 Sidebar UI、消息桥接、诊断读取、终端接入、补丁预览、审批交互、密钥存储

随着产品演进，未来可继续扩展：

- CLI 客户端
- Electron 桌面端
- 更多模型适配器
- 更多工具与权限策略

但这些扩展都不应该破坏第一版建立的核心边界。

### English

The first version is easiest to understand as two layers:

- Agent Core: responsible for runtime orchestration, planning, execution, tool scheduling, context governance, and model adaptation
- VS Code Adapter: responsible for Sidebar UI, message bridging, diagnostics access, terminal integration, diff preview, approval interaction, and secret storage

As the product evolves, it can later expand to:

- a CLI client
- an Electron desktop client
- more model adapters
- more tools and permission strategies

None of these future expansions should break the core boundary established in the first version.
